# 二、需求分析

本章围绕 iDrug——多场景 DTI 预测与个性化药物推荐辅助平台的目标用户、核心功能和业务约束展开需求分析。平台面向药物研发研究人员、生物信息方向师生及数据分析用户，围绕“预测—查询—排序—展示”的核心流程，提供多场景 DTI 预测、个性化药物推荐、结构查询与可视化、数据浏览以及用户结果管理等功能。

## 1. 用例模型

### 1.1 用例概述

| 用例编号 | 用例名称 | 级别 | 主要参与者 | 简要说明 |
|---|---|---|---|---|
| UC01 | 用户注册与登录 | 用户目标 | 平台用户 | 用户完成账号注册、登录、身份校验，为结果保存和历史记录管理提供身份基础。 |
| UC02 | 多场景 DTI 预测 | 用户目标 | 研究人员、学生、数据分析用户 | 用户输入药物与靶点信息，选择任务类型，系统调用对应模型并返回预测结果。 |
| UC03 | 个性化药物推荐重排序 | 用户目标 | 研究人员、学生、数据分析用户 | 系统结合 DTI 预测结果、用户画像和药物属性，对候选药物进行二次排序。 |
| UC04 | 药物、蛋白质与复合物结构查询 | 用户目标 | 研究人员、学生、数据分析用户 | 用户按药物、蛋白或复合物条件查询结构信息，并查看三维可视化结果。 |
| UC05 | 数据浏览与本地知识库查看 | 子功能 | 平台用户 | 用户浏览平台整理的本地结构化数据，为预测、推荐和查询提供数据支撑。 |
| UC06 | 预测结果保存与历史管理 | 用户目标 | 已登录用户 | 用户保存预测或推荐结果，在用户中心查看、复用和删除历史记录。 |

| 发生频率 | 用户首次使用、重新进入系统或需要保存结果时发生，频率较高。 |
| 杂项 | 后续应进一步确认密码加密存储、会话过期策略和权限控制策略。 |

### 1.3 UC02_多场景 DTI 预测

| 项目 | 内容 |
|---|---|
| 用例编号与名称 | UC02_多场景 DTI 预测 |
| 范围 | iDrug——多场景 DTI 预测与个性化药物推荐辅助平台 |
| 级别 | 用户目标 |
| 主要参与者 | 药物研发研究人员、生物信息方向师生、数据分析用户 |
| 涉众及其关注点 | 研究人员：希望基于药物和靶点输入快速获得 DTI 预测结果。<br>学生用户：希望通过统一页面理解不同 DTI 任务的输入输出。<br>系统维护者：关注模型调用是否稳定、模型依赖环境是否隔离。 |
| 前置条件 | 平台后端已启动；模型推理脚本和必要权重文件已部署；用户能够提供药物与蛋白质输入信息。 |
| 成功保证 | 系统根据任务类型调用合适模型，返回预测分数、预测标签或候选结果列表，并在页面上清晰展示。 |
| 主成功场景 | 1. 用户进入多场景预测页面。<br>2. 用户选择任务类型，如药物重定位、新靶标筛选、新药评估或亲和力预测。<br>3. 用户输入药物信息，如 SMILES、CID 或候选药物列表。<br>4. 用户输入靶点信息，如蛋白质序列、PDB ID、UniProt ID 或候选靶标列表。<br>5. 前端提交预测请求。<br>6. 后端根据任务类型执行任务—模型匹配。<br>7. 系统调用 HyperAttentionDTI、PSICHIC 或 DeepDTA 等模型进行推理。<br>8. 后端解析模型输出并统一封装结果。<br>9. 前端展示预测分数、排序结果和任务状态。 |
| 扩展 | 2a. 用户未选择任务类型：系统提示选择预测任务。<br>3a. 药物输入格式错误：系统提示输入合法 SMILES、CID 或药物名称。<br>4a. 蛋白质输入为空或格式不合法：系统提示补充靶点信息。<br>7a. 模型权重缺失或推理环境不可用：系统返回模型不可用说明。<br>7b. 模型运行超时：系统提示任务失败，并保留输入信息便于重新提交。 |
| 特殊需求 | 不同模型依赖环境应隔离部署；预测结果应有统一的数据结构；长时间推理应有加载状态和错误提示；结果仅作为科研辅助分析，不作为临床诊疗结论。 |
| 技术与数据变元素 | 输入数据包括 SMILES、蛋白质序列、PDB ID、UniProt ID、CSV 批量输入等；模型输出包括预测概率、亲和力分数、分类标签和排序列表；后端通过 Django 统一调度不同 Python 推理环境。 |
| 发生频率 | 平台核心功能，用户每次开展 DTI 分析时触发，频率高。 |
| 杂项 | 不同模型的指标和适用场景不同，页面说明中应避免将不同任务结果直接作不恰当比较。 |

### 1.4 UC03_个性化药物推荐重排序

| 项目 | 内容 |
|---|---|
| 用例编号与名称 | UC03_个性化药物推荐重排序 |
| 范围 | iDrug——多场景 DTI 预测与个性化药物推荐辅助平台 |
| 级别 | 用户目标 |
| 主要参与者 | 研究人员、生物信息方向师生、数据分析用户 |
| 涉众及其关注点 | 平台用户：希望系统不仅给出模型分数，还能结合疾病、过敏和医保等条件组织候选结果。<br>药物研发或教学人员：关注推荐依据是否可解释。<br>系统维护者：关注药物属性数据、疾病标签和过敏标签是否可维护。 |
| 前置条件 | 已存在候选药物与靶点输入；基础 DTI 预测可执行；本地药物个性化元数据、疾病标签和过敏标签可读取。 |
| 成功保证 | 系统输出个性化排序后的候选药物列表，并展示基础预测分数、综合评分、排序变化和推荐理由。 |
| 主成功场景 | 1. 用户进入个性化推荐页面。<br>2. 用户输入药物、靶点及候选条件。<br>3. 用户填写画像信息，如年龄、性别、疾病史、过敏史和医保偏好。<br>4. 系统先执行基础 DTI 预测，得到候选药物基础分数。<br>5. 系统读取药物属性、适应症、禁忌症、医保属性等本地知识数据。<br>6. 系统按照“基础预测分数 + 疾病匹配加分 - 风险惩罚 + 偏好权重”的思路计算综合得分。<br>7. 系统生成重排序结果。<br>8. 前端展示推荐排名、分数变化和解释性说明。 |
| 扩展 | 3a. 用户未填写画像：系统可退化为仅按基础 DTI 分数排序。<br>3b. 用户填写过敏史与候选药物存在潜在冲突：系统降低候选药物排序并给出风险提示。<br>5a. 某候选药物缺少属性数据：系统保留基础预测结果，并提示解释信息有限。<br>6a. 数据格式异常：系统跳过异常字段并记录错误，避免影响全部结果生成。 |
| 特殊需求 | 推荐理由应可读、可解释；涉及疾病、过敏、医保等信息时应提示其仅用于课程项目和科研辅助分析；不应替代医生诊断或处方建议。 |
| 技术与数据变元素 | 输入包括药物列表、靶点信息、年龄、性别、疾病史、过敏史、医保偏好；数据来源包括 DrugBank 整理数据、本地 CSV、疾病标签字典和过敏标签字典；输出为 JSON 排序列表。 |
| 发生频率 | 用户进行药物推荐、药物重定位分析或课程演示时发生，频率中高。 |
| 杂项 | 后续可继续完善知识库覆盖范围，并引入更严格的医学风险校验规则。 |

## 2. 补充性规格说明

| 类别 | 要求 |
|---|---|
| 性能 | 普通页面请求应在较短时间内响应；数据浏览应采用分页或摘要返回；模型推理任务允许较长运行时间，但应提供加载状态和失败提示。 |
| 可用性 | 页面应提供清晰导航，支持首页、预测、推荐、查询、数据浏览、教程和用户中心之间快速跳转；输入错误时应给出可理解的提示。 |
| 可靠性 | 外部 API 不可用时，系统应尽量使用本地缓存或返回可解释的错误信息；单个模型调用失败不应影响其他页面的基本访问。 |
| 安全性 | 数据库密码、API Key、模型权重路径等敏感信息不得硬编码到公开仓库；用户身份信息和历史记录应按用户隔离；日志中不应输出敏感字段。 |
| 兼容性 | 系统面向主流桌面浏览器；前端由 HTML、CSS、JavaScript 构成；后端基于 Django；数据库使用 MySQL；模型可运行在独立 Python 环境中。 |
| 可维护性 | 后端应按预测服务、推荐排序、查询展示、用户管理等模块划分；模型调用应通过统一调度接口封装；本地数据文件应有字段说明。 |
| 可扩展性 | 后续应支持更多 DTI 模型、更多公开数据库接口、更多药物属性标签和更完整的结果导出功能。 |
| 数据要求 | 药物输入应支持 SMILES、CID 或名称等方式；蛋白输入应支持序列、PDB ID 或 UniProt ID；本地数据应保持字段命名一致。 |
| 可解释性 | 预测和推荐结果应尽量展示分数来源、排序原因、疾病匹配、风险惩罚和数据来源，避免只给出不可解释的数值。 |
| 医学边界 | 平台输出仅用于课程展示、科研辅助和教学实验，不构成临床诊断、处方或用药建议。 |

## 4. 设想

核心愿景：建设一个集多场景 DTI 预测、个性化药物推荐、多源数据查询、结构可视化和结果管理于一体的 Web 辅助平台，使用户能够通过统一页面完成从药物与靶点输入、模型预测、结果排序、结构解释到历史保存的完整流程。

关键目标包括：

1. 支持多场景预测。平台不局限于单一 DTI 模型，而是根据药物重定位、新靶标筛选、新药评估和亲和力预测等任务自动匹配合适模型。
2. 提供可解释的推荐结果。平台在基础预测分数上引入疾病匹配、过敏风险、医保偏好等因素，帮助用户理解候选药物排序变化。
3. 融合多源数据服务。平台整合本地结构化数据、MySQL 业务数据以及 PubChem、PDB、DrugBank 等公开数据资源，提高数据使用效率。
4. 提升结果表达能力。平台通过结构查询与三维可视化，让用户从药物分子、蛋白结构和复合物层面辅助理解预测结果。
5. 形成完整 Web 应用闭环。平台支持注册登录、结果保存、历史记录查看和删除，使系统从算法演示进一步升级为可使用的软件系统。

当前设想主要面向课程展示、科研辅助分析和教学实验场景。未来可进一步拓展更多公开数据集、模型解释方法、结果导出能力、团队协作能力和云端部署能力。

## 5. 业务规则

1. 用户必须完成登录后，才能保存预测结果、查看个人历史记录或删除个人记录。
2. 注册时用户名应保持唯一；登录时用户名和密码必须同时满足校验条件。
3. 平台输出的预测、推荐和风险提示仅用于科研辅助、教学实验和课程展示，不得作为临床诊断或处方依据。
4. 多场景预测必须根据任务类型进行模型匹配：药物重定位、新靶标筛选、新药评估和亲和力预测应调用对应模型或模型组合。
5. 药物输入必须经过格式检查，系统应尽量支持 SMILES、CID、药物名称等常见输入形式；无法解析时应提示用户修改输入。
6. 蛋白质或靶点输入必须经过格式检查，系统应尽量支持蛋白质序列、PDB ID、UniProt ID 等形式；无法解析时应给出明确提示。
7. 个性化推荐应优先保留基础 DTI 预测结果，再结合疾病史、过敏史、医保偏好和药物属性进行综合排序。
8. 当候选药物与用户过敏史或风险标签存在冲突时，系统应降低其排序并给出风险说明。
9. 当药物缺少个性化元数据时，系统可保留基础预测分数，但应提示推荐解释信息不足。
10. 结构查询应优先使用本地缓存或本地结构化数据；本地数据不足时再调用 PubChem、PDB 等外部接口。
11. 外部接口调用失败时，系统不得直接崩溃，应返回可解释的错误提示或展示本地可用信息。
12. 数据浏览页面应采用分页或摘要机制，避免一次性加载大型数据导致页面卡顿。
13. 用户保存的历史记录应与用户身份绑定，普通用户不得查看或删除其他用户的私有记录。
14. 公开 demo 记录可用于课程展示和功能演示，但应避免被普通用户误删。
15. 模型权重、大型数据集、数据库密码、API Key、服务器账号等不得提交到公开 GitHub 仓库。
16. 系统日志不应记录明文密码、API Key、Token 或完整敏感用户信息。
17. 前端页面应对关键操作提供反馈，包括预测中、保存成功、保存失败、删除成功和输入异常等状态。
18. 数据库初始化应通过 Django migrations 或说明文档完成，避免依赖不可追踪的手工建表步骤。
19. 模型运行环境可按模型分别隔离，后端通过统一调度接口调用，避免依赖冲突影响系统稳定性。
20. 后续功能扩展应保持“预测—查询—排序—展示—保存”的主流程一致，新增模型或数据源需补充说明其适用场景和输入输出格式。
